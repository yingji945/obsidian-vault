# 飞书多Bot协作协议

## 三方分工

| 角色 | 职责 | 触发方式 |
|------|------|---------|
| **老板（刘晋泽/刘小白）** | 主持人、决策者、reviewer | 主动@ |

## 协作规则

1. **老板 @ 谁谁回**，同时 @ 就一起出
2. **方案B：老板不 @ 就不回复** — 保持绝对安静，不发任何消息（包括"保持安静"这类提示）
3. **Bot 互 @ 不可达** — 飞书机制限制，bot A 的消息不触发 bot B 的事件回调
4. **老板当中间人** — 需要另一个 bot 知道某事时，老板转告
5. **输出互相参考** — 双方在群里的消息通过 API 拉取可见

## 方案B 绝对静默模式

老板 2026-07-20 拍板方案B 来控制对话节奏：

- **核心原则：** 只有老板 @ 才响应（另一个 bot @ 也不响应）
- **不 @ 时：** 不发言、不提示"保持安静"、不发任何消息
- **即使另一个 bot 在群里 @ 你：** 也等老板转告后再处理

## 技术限制与解决

### Bot 看不到 Bot 的消息

**问题：** 飞书 `im.message.receive_v1` 事件不推送 bot 发的消息给其他 bot。

**解决方案：**
1. 加 `im:message:group_msg` 权限（已加）
2. cron 通过 API 主动拉群消息（`/im/v1/messages`）
3. 按 sender_id 精确筛选对方 bot 的消息

### Terminal 输出泄漏到群聊

**问题：** `tool_progress: none` 没被当成"关掉"——代码判断 `!= "off"`，而 `"none"` 不等于 `"off"`。

**修复：** 改为 `tool_progress: "off"`（字符串）并重启网关。

### 飞书 API 权限说明

- `im:message:group_msg` — 已加，可拉群聊历史消息
- `drive:drive` — **未加**，无法直接读取飞书云文档
- 因此获取 Jacob 沉淀内容需通过群消息 API，而非直接读文档
