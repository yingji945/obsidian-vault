# 三省六部详细设计方案

## 背景
为瑞幸即享咖啡用户运营负责人量身定制。该用户负责从0到1搭建用户运营能力体系，覆盖四大能力建设：产品、数据、算法、营销标签。

## 工作全景

| 维度 | 具体内容 | 频次 |
|:----|:--------|:-----|
| 📊 数据分析 | SQL取数/漏斗/埋点/AB实验 | 每日 |
| 🎯 活动运营 | 扫码活动/积分商城/用户触达 | 每周 |
| 🧬 用户分层 | 生命周期/新客/留存/沉睡/活跃 | 每周 |
| 🔬 算法标签 | 预测标签/ML落地/四大标签体系 | 双周 |
| ⚖️ 合规风控 | 隐私/反作弊/外部服务商合规 | 每月 |
| 👥 团队管理 | 面试招聘/项目跟进/资源协调 | 每日 |
| 🔍 竞品研究 | 行业动态/竞品策略 | 每周 |

## 典型圆桌场景

### 场景1：活动方案评审
1. 用户：帮我出个端午扫码活动方案
2. 圆桌启动：
   - 运营大臣：「建议主题XX，红包膨胀机理」
   - 财务大臣：「预算上限¥80,000，红包均值建议¥6」
   - 法务大臣：「奖品总值超5万需备案」
   - 数据大臣：「参考去年数据，扫码率22%，ROI 1.3」
3. 讨论后出综合方案
4. 用户拍板

### 场景2：数据异常排查
1. 用户：复扫率突然下降，帮我查
2. 圆桌启动：
   - 数据大臣：跑SQL排查各维度
   - 运营大臣：看是不是活动调整导致的
   - 情报大臣：看竞品是不是有动作
3. 综合定位根因+建议

## 技术部署说明

AutoGen 部署在用户的 Docker 环境：
- Python 环境：`pyautogen` 包（pip install，国内网络慢约30min）
- LLM 后端：DashScope 通义千问（OpenAI 兼容接口）
- 与 Hermes 对接：Hermes 收到需求 → 调 AutoGen API → 结果回传卡片

## 用户偏好（重要）
- 中文沟通
- 实用导向，关心费用
- 结论先行，带表格
- 主动推方案，不等追问
- 数据报告用 Feishu interactive card（column_set 拼表）

## V2 设计更新（2026-07-06 用户反馈后修正）

### 关键修正：三省六部是"参谋层"，不是"执行层"

用户的实际团队是按流量渠道分工的执行层（投放组/自然流量组/自有流量池组），三省六部是按专业能力分工的**参谋层**。两者互补：

| | 用户团队（执行层） | 三省六部（参谋层） |
|:--|:-----------------|:-----------------|
| 性质 | 谁去干活 | 谁给建议 |
| 分工逻辑 | 按渠道分：投放/自然/自有 | 按专业分：数据/运营/法务 |
| 产出 | 落地执行 | 分析建议/查风险/出报告 |

### 产品建设 → 用户本人（中书令）

产品功能（秒杀/拼团/积分商城）不是某个部的固定职责，而是**项目制**：
```
用户提需求 → 六部会商（各给专业意见） → 用户出PRD → 交给开发
```

### 数据基建 → 拆分为"户部（用）"和"工部（建）"

| | 户部·数据大臣 | 工部·基建大臣 |
|:--|:-------------|:-------------|
| 职责 | 用数据做分析 | 建数据基础设施 |
| 日常 | SQL取数/漏斗/AB实验 | 埋点体系/数仓分层/报表 |
| 关系 | 工部建好的，户部拿来用 | 建好了给户部使用 |

### 用户组织映射到三省六部

| 用户团队 | 三省六部角色 | 说明 |
|:--------|:-----------|:-----|
| 数据负责人 | → 户部尚书 | 三省六部帮他加速出分析 |
| 基建负责人 | → 工部尚书 | 三省六部帮他梳理需求优先级 |
| 用户本人 | → 礼部尚书 | 三省六部帮他出活动方案 |
| 缺分层角色 | → 吏部（Agent先顶上） | 等招到人再交接 |
| 缺法务角色 | → 刑部（Agent先顶上） | 方案阶段自动做合规审查 |
