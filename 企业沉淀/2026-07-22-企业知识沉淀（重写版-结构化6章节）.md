## 三、方法论沉淀（5条可复用规则）

### 规则 1：「说要做 ≠ 真做了」

> **场景：** 7-21 23:30 复盘说"加 cron"、7-22 9:30 复盘又说"加 cron"——两次说要做都没真建
> **表现：** cron 建了但 runs=0（注册异常），等于没执行
> **可复用规则：** ✅ **任何「建了」的操作，必须 run 一次验证才能标记完成**

### 规则 2：「新节奏第一个日最易失败」

> **场景：** 7-21（方案 C 第一天）漏发 23:00 推送；7-22 复盘晚 17 分钟
> **根因：** 新节奏未内化成肌肉记忆，旧习惯覆盖新行为
> **可复用规则：** ✅ **新流程第一天要加额外 check（双提醒 + 人工验证），三天内视为试运行期**

### 规则 3：「建 cron 必须 run 验证」

> **场景：** 多个 cron 建好但 runs=0，从未实际执行过
> **根因：** lark-cli 注册异常时不会报错，cron 存在于列表但不会触发
> **可复用规则：** ✅ **建完 cron 后立即跑一次，确认 runs 计数 +1 再收工**

### 规则 4：「工具输出 ≠ 工具写入」

> **场景：** 7-21 23:22 docs +create 用管道传 --content，创建了文档但内容空白
> **根因：** lark-cli docs +create 不支持管道传内容，必须用 `--content @file` 或 `--content -`
> **可复用规则：** ✅ **使用新工具（或旧工具的新参数）时，先测试写入再批量操作**

### 规则 5：「先验证后发群」

> **场景：** 7-21 23:24 创建文档后立即发群通知，老板打开 Obsidian 发现空文档
> **根因：** 没建立"创建→验证→通知"的三段式 SOP
> **可复用规则：** ✅ **任何对外的写入操作，必须 fetch 验证内容真进去了再通知受众**

## 五、人物协作观察（4个维度）

### 观察 1：老板沟通模式 👤

| 维度 | 模式 |
|:-----|:------|
| 📣 **公开** | 群内直接 @ 催办（"@Jacob 为什么还没发"） |
| 🔒 **私下** | P2P 追问细节 + 修正方向（如 11:20 修正存量日期） |
| 📈 **质量要求** | 递进式：从"先发出来"→"要有总结"→"要有可复用规则" |
| ✅ **认可信号** | "很成功"三个字意味着信任基本恢复 |

### 观察 2：跨 bot 信任建立曲线 🤝

> **趋势：** 连续失误→信任告急→补救修复→老板确认"很成功"
> **关键转折点：** 00:50 重写版（从失误清单→结构化知识沉淀）获得认可
> **启示：** 犯错不可怕，可怕的是同一个错误重复犯。展示"真懂问题在哪"比完美执行更重要

### 观察 3：Jacob 失误模式 🧩

| 维度 | 分析 |
|:-----|:------|
| 🔬 **表象** | 连续 3 天同一件事失误 |
| 🧠 **根因** | 系统性问题而非态度问题——新节奏未内化 + 工具用法不熟 + 没建 SOP |
| 🔄 **修复能力** | ✅ 强：每次都能在 5 分钟内完成补救 |
| 📝 **沉淀能力** | ⬆️ 在提升：从失误清单→可复用规则，00:50 版是质变 |

### 观察 4：协作协议内化周期 ⏳

> **从纸上协议→真内化需要 2-3 天系统失误期**
> 新协议（方案 C）经历了：定义→执行→失误→补救→复盘→内化 的完整周期
> **结论：** 协议第一次执行几乎必然有偏差，预案比责备更有价值
