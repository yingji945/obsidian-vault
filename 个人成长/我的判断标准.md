# 我的判断标准

> 这是我判断一件事「好不好」的原则。不是知识，是品味。
> 会随着经验持续更新。

---

## 一、什么样的 SQL / 数据分析算「好」

### 流程标准

一条好的 SQL 查询，在我的标准里：

```
   先给结论（一行+颜色信号）
   ↓
   数据带对比（不孤立的数字）
   ↓
   写明风险点
   ↓
   给出下一步
   ↓
   代码本身干净、能跑、能复用
```

### 代码标准

| 维度 | 好的 | 不好的 |
|:----|:----|:------|
| 性能 | 拆 OR 为 UNION ALL，不加无谓的 JSON 解析，小表广播 | 大表全扫、OR 里混 JSON、不用 broadcast hint |
| 可读 | CTE 分段清晰，别名有含义 | 一坨 WHERE + 20个 OR |
| 可复用 | 日期边界用变量风格，方便改范围 | 硬编码 `date_sub(current_date(),1)` 藏在里面 |
| 安全 | `NULL NOT IN` 加 `is null or` 保护 | 忘记 NULL 语义，净流量全丢了 |

### 关键原则

1. **不加无谓的复杂度**。能用一个 JOIN 解决别用两个，能用 IN 别用 OR。
2. **「好分析」= 能推动决策的分析，不是数据最全的分析**。
3. **结论先行，永远带对比，永远有下一步。** 这三个缺一个就是不合格。

## 二、什么样的 AI 协作模式算「好」

### 核心原则

> **AI 做大脑做编排，成熟工具做手脚。不要什么活都自己揽。**

这句话翻译成我的标准：

| 好的设计 | 不好的设计 |
|:--------|:----------|
| Hermes 调三省六部做决策，调 API 发卡片 | 我自己写死所有逻辑 |
| 用飞书原生 Bot 拉文档，Hermes 整理入库 | Hermes 自己去解析飞书文档格式 |
| 三省六部讨论出结论，Kanban 记录，Web 旁路展示 | 三省六部自己记录+展示+通知 |
| 专业工具干专业活（Working Copy 同步手机） | 我写个脚本自己做同步 |

### 更本质的一句话

> **你在设计的是「人机协作的组织架构」，不是在使用工具。**

三省六部、大脑+手脚、Hermes+Jacob 分工——本质上都是在搭建一个**组织**，再把自己放进最合适的位置。

## 三、什么样的「体系」算「好」

| 维度 | 好的体系 | 不好的体系 |
|:----|:--------|:----------|
| 清晰 | 每个角色/层级/位置职责明确 | 一个人什么都要做 |
| 边界 | 知道自己不能做什么 | 觉得自己什么都能做 |
| 可进化 | 有反馈机制，能迭代 | 建完就固定了 |
| 容错 | 有人犯错也不会全崩 | 一个人掉链子整个系统停 |

### 从三省六部学到的

三省六部最让我满意的不是它可以讨论问题——是它设计了一个**角色分界**。数据大臣只管数据和 SQL，运营大臣只管业务判断，法务大臣只管合规。

**每个人只做自己最擅长的事，比一个人强行全能有效得多。**
这个原则也适用于我自己——不是所有事都要自己干。

---

## 四、这个笔记的用途

1. **写 SQL 前，拿第一条过一遍** —— 有没有先给结论？有没有带对比？
2. **设计新的 AI 工作流时，拿第二条过一遍** —— 是不是大脑+手脚？边界清楚吗？
3. **建新的体系时，拿第三条过一遍** —— 角色清晰吗？有容错吗？能迭代吗？

> 这些标准不是固化的，它们应该随着我的工作经验变。
> 下次重读时，如果觉得哪条不对了，就改——那就是我在成长。
