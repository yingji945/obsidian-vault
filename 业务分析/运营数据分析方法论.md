# 运营数据分析方法论

## 漏斗分析原则

### 链路归因
分析用户行为链路时，**明确每一步的约束条件**：

```sql
-- 模板：三层漏斗，同日期约束
WITH step1 AS (  -- 第一步：入口
  SELECT user_id, dt
  FROM table_a
  WHERE ...
),
step2 AS (  -- 第二步：行为
  SELECT user_id, dt
  FROM table_b
  WHERE ...
),
step3 AS (  -- 第三步：转化
  SELECT mem_id, dt, income
  FROM table_c
  WHERE ...
)
SELECT
  COUNT(DISTINCT s1.user_id) AS step1_uv,
  COUNT(DISTINCT s2.user_id) AS step2_uv,
  COUNT(DISTINCT s3.mem_id)  AS step3_uv,
  COALESCE(SUM(s3.income), 0) AS total_income
FROM step1 s1
LEFT JOIN step2 s2 ON s1.user_id = s2.user_id AND s1.dt = s2.dt
LEFT JOIN step3 s3 ON s2.user_id = s3.mem_id AND s2.dt = s3.dt
```

### 常见坑点

| 问题 | 后果 | 修复 |
|:----|:-----|:-----|
| `LEFT JOIN` 只关联用户不关联日期 | 同用户不同天的数据交叉，金额翻倍 | 关联时加上 `AND date = date` |
| CTE 内预关联下游表 | 下游表数据被上游多行重复匹配 | 只在最终 SELECT 时 join |
| OR 条件包含 JSON 解析 | 全表触发 JSON 解析，极慢 | 拆成 UNION ALL |
| `prop_query_source IS NULL` 兜住自然流量 | 但未归因的投放流量也被兜入 | 看业务场景决定是否保留 |
